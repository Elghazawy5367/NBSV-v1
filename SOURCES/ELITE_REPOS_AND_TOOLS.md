## SECTION 2 — SELF-HEALING REPO SETUP

Self-healing means the repo improves itself over time without manual maintenance effort. These are all GitHub-native features, browser-configurable.

### Auto-Watch: Stay Current Without Effort

**GitHub Watch (your most important passive tool):**

For each elite repo you identify:
1. Go to the repo
2. Click **Watch** → **All Activity**
3. GitHub emails you whenever the repo updates

Applied to 10–15 well-chosen repos = a daily intelligence briefing that arrives in your inbox. No scanning required.

**Recommended Watch List (set up once):**
- 3 laser cutting design file repos → you see every new design file immediately
- 2 trending product tracker repos → published data arrives in your inbox
- 2 awesome lists in your categories → curators update for you
- 1 Kickstarter analytics repo → campaign data published periodically
- 1 Chinese platform trend output repo → signals arrive without daily Douyin scanning

**GitHub Stars as a curated reading list:**
- Star anything worth revisiting
- Stars are public — see what other product researchers star: `github.com/[username]?tab=stars`
- Search starred repos: `github.com/search?q=starred:username+laser+cut`

### Auto-Sync: Keep Forks Current

When you fork a design file repo, the original continues improving. Sync your fork to get updates:

1. Go to your forked repo
2. GitHub shows: "This branch is N commits behind original"
3. Click **Sync fork** → **Update branch**
4. Your fork now has all new design files the original published

**Do this monthly.** Takes 10 seconds per fork.

### Branch Protection: Protect Your KB

Prevents accidental overwrites of the authoritative KB:

1. Repo → Settings → Branches
2. Branch name: `main`
3. Enable: **Require a pull request before merging**
4. Enable: **Require approvals** → set to 1 (you approve your own changes)

Effect: You cannot accidentally overwrite KB v4 by dragging a file to the wrong place. All KB changes go through a PR (which uses your PR template checklist).

### GitHub Wiki: Self-Populating Knowledge Layer

The Wiki is separate from the main repo and grows automatically as you add to it. Set up these pages once, then they compound:

**Wiki pages to create:**
1. `Failure-Pattern-Catalog` — paste failure log entries monthly. Over time = searchable rejection intelligence.
2. `Cultural-Fit-Heuristics` — every strong Egyptian behavior answer that produced a PASS. Compounds.
3. `Workshop-Performance-Notes` — patterns across workshop types, materials, timelines. (No names — area + capability only.)
4. `Seasonal-Retrospectives` — one page per season completed. What amplified, what didn't, what arrived late.
5. `Kill-Signal-Frequency` — running tally of which kill signals dominate. Updates after each failure log session.

**Self-healing mechanism:** Feed the Wiki to Claude/NotebookLM each season: *"Based on these patterns, what should I scan for more? Less? What behavior type keeps producing PASS?"*

## SECTION 3 — CODE MIRRORING STRATEGY

Code mirroring = keeping local copies of external repos that provide value, with automatic sync. You never lose access to a design file because the original was deleted.

### What to Mirror (Fork)

**Rule:** Mirror anything you might want to produce from. Fork → your copy is permanent even if original is deleted.

| Category | What to Fork | How to Use |
|---|---|---|
| Laser cut design files | Any repo with `.svg` or `.dxf` organizer/storage designs | Download files → send to workshop → get COGS quote |
| Product design templates | Workshop spec templates, cut list templates | Copy into your `CANDIDATES/` folder, customize |
| Awesome lists | `awesome-product-design`, `awesome-laser-cutting` | Reference — never goes out of date if you fork it |
| Trend analysis outputs | Published CSV/JSON files from scraper repos | Read the data directly — no code needed |

### Forking Protocol (Browser Only)

1. Find a repo to mirror
2. Click **Fork** (top right)
3. Select your account as destination
4. Done — your fork is at `github.com/[your-username]/[repo-name]`
5. Add a note in the repo's Description field about why you forked it

**Monthly:** Go to your Forks list → Sync each fork → Check for new files.

### Design File Processing Pipeline (No Code)

When you find a design file repo with usable `.svg` or `.dxf` files:

```
1. Fork the repo (browser)
   ↓
2. Open the .svg or .dxf file in GitHub browser viewer
   ↓
3. Download the raw file (click Raw → save the page)
   ↓
4. Send file to workshop via WhatsApp: "Can you cut this? Price at 30 units?"
   ↓
5. If COGS ≤80 EGP — run 3-question filter on the design concept
   ↓
6. If PASS — create GitHub Issue with the design as source
```
